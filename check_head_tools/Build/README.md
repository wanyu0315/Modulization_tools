### 冗余头文件自动化编译验证工具链

#### 📌 简介

本工具链用于对静态扫描输出的“疑似冗余头文件”进行**真实的物理编译验证**。 由于 C++ 存在复杂的隐式依赖、宏定义和预编译头（PCH）机制，静态扫描的结果可能存在误报。本工具链通过自动化调用底层 `MSBuild.exe`，提供从“极速增量快筛”到“严苛四象限重构”的多维度验证方案，确保头文件的清理绝对安全，实现 **100% 编译绿灯**。

------

#### ⚙️ 核心脚本说明

##### 1. `super_build.py` (全量四象限严苛验证)

- **定位：** 终极真理，适合周末或夜间 (Nightly) 的无人值守全量测试。

- **工作原理：** 采用单步串行模式。对 CSV 中的每一个头文件，依次执行 `Build(Debug)` -> `Build(Production)` -> `Rebuild(Debug)` -> `Rebuild(Production)`。

- **特性：** 包含 Fail-Fast（快速失败）机制，一旦某一环节报错立即标记为 `N` 并跳过后续极其耗时的重构。

- **用法示例：**

  ```
  python super_build.py --csv output.csv -s "D:\ZW3D\cad.sln" --msbuild "C:\...\MSBuild.exe"
  ```

##### 2. `fast_build.py` (增量快筛漏斗)

**定位：** 日常开发的“第一道防线”，用于快速剔除明显的误报（如宏定义缺失）。

**工作原理：** 单步串行模式。仅执行耗时较短的增量 `Build (Debug)` 和 `Build (Release)`。

**输出：** 将通过了双端增量编译的“幸存者”单独输出到一份全新的 `build_passed_xxx.csv` 文件中，供下一阶段使用。

**⚙️ 参数说明**

| **参数**            | **默认值**          | **说明**                                                     |
| ------------------- | ------------------- | ------------------------------------------------------------ |
| `--csv`             | `output.csv`        | 输入待验证的 CSV 文件路径                                    |
| `-s`, `--sln`       | **(必填)**          | 解决方案 (`.sln`) 的绝对路径，即编译图纸                     |
| `--msbuild`         | *(VS2022 默认路径)* | 使用的本地编译器的绝对路径                                   |
| `--timeout-build`   | `600`               | 单次构建的超时时间（秒）                                     |
| `--module`          | *(空)*              | 只测试指定的子模块（如：`cad`），为空则测试大表中的全部记录  |
| `--use-project-opt` | 开启                | 项目级寻址优化，开启时尝试根据当前正在测试的文件路径，直接找到它所属的单个子项目构建文件（即 `.vcxproj`）并用它进行构建，**完全跳过**了对整个工程依赖树的扫描 |
| `--no-project-opt`  | 未开启              | 直接输入`--no-project-opt`，脚本会将顶级解决方案文件完整地提交给 MSBuild。 |

**🚀 用法示例**

**🟢 场景一：只测试“单个子模块”（例如：只清理 cad）**

- **适用时机：** 日常开发，只负责特定模块，追求极致的验证速度。
- **执行逻辑：** 脚本只修改大表中的 `cad` 代码，且 MSBuild 只利用叶子节点优化增量编译 `cad.sln` (或 `cad.vcxproj`)。
- **命令特点：** 速度最快。编译器聚焦单一模块，完全不理会整个项目的其他海量文件。

> ```
> python fast_build.py --csv union_all.csv -s "D:\ZW3D\out\build\cad\cad.sln" --module "cad"
> ```

**🟡 场景二：同时测试“若干个子模块”（例如：清理 cad 和 geometry）**

- **适用时机：** 负责几个相互关联的底层模块，想要一次性将它们清理干净。
- **执行逻辑：** 脚本同时揪出这几个模块的代码进行修改。注意：因涉及跨模块，`-s` 必须使用包含了它们的上级顶层 `.sln`，否则编译器无法解析跨模块依赖。
- **命令特点：** 速度中等。多个模块用逗号紧密相连（中间无空格），启动了全局预检查机制。

> ```
> python fast_build.py --csv union_all.csv -s "D:\ZW3D\out\build\zw3d.sln" --module "cad,geometry"
> ```

**🔴 场景三：测试“整个庞大项目”（全局大清洗）**

- **适用时机：** 版本发布前夕，或利用极高性能服务器/夜间挂机运行，进行彻底的代码瘦身。
- **执行逻辑：** 脚本对大表里的所有文件无差别开火（不加 `--module` 限制），并让编译器审查全站 `zw3d.sln`。
- **命令特点：** 速度最慢，但绝对安全、零死角。所有隐含的跨模块依赖都会在此模式下现出原形。

> ```
> python fast_build.py --csv union_all.csv -s "D:\ZW3D\out\build\zw3d.sln"
> ```

##### 3. `batch_rebuild.py` 

**定位：** 提交代码至主干分支前的“终极防线”与“精准定责工具”。用于在全域 Rebuild 极其严苛的约束下，处理跨模块隐式依赖冲突，并输出最大化的安全重构子集。

**工作原理（二分逼近 Delta Debugging）：** 放弃了传统的单步串行（耗时过长）与全量梭哈（容错率为零）策略。

1. 脚本首先将 CSV 列表中的所有待测目标一次性注释，并发起全局 Rebuild。
2. 若成功，直接全量入库。
3. 若失败，脚本会自动将待测列表对半切割（二分），并分别针对子区间进行 Rebuild 探测。
4. 算法递归向下拆解，以 $O(K \log_2 N)$ 的时间复杂度精准剥离导致编译中断的 $K$ 个核心依赖节点，并将其余安全文件自动合并保留。

**输出：** 1. **物理层：** 所有被证实安全的冗余头文件，其在源码中的 `//` 注释将被永久保留；引发冲突的文件将被安全回滚。

2. **数据层：** 生成一份全新的 `rebuild_bisection_result_xxx.csv`。针对参与本次测试的条目，追加 `Rebuild_Passed` 状态列（`Y` 代表验证通过且已注释，`N` 代表触犯依赖规则已还原，`-` 代表未参与本轮测试）。

**参数说明**

| **参数**      | **默认值**          | **说明**                                                     |
| ------------- | ------------------- | ------------------------------------------------------------ |
| `--csv`       | **(必填)**          | 输入阶段一（`fast_build.py`）输出的过滤后名单                |
| `-s`, `--sln` | **(必填)**          | 顶层解决方案 (`.sln`) 的绝对路径，用于解析全域依赖图         |
| `--msbuild`   | *(VS2022 默认路径)* | 本地 MSBuild 编译器的绝对路径                                |
| `--timeout`   | `7200`              | 单次 Rebuild 的最大容忍耗时（秒）。超时将触发强制中断与环境清理 |
| `--module`    | *(空)*              | 限定执行二分测试的子模块范围（支持逗号分隔，如 `cad,geometry`）。留空则执行全域并发测试 |
| `--max-cpu-count` |  max(1, min(cpu_total - 4, 16))  | 本地 MSBuild 编译时CPU核的最大并行度，建议最大核心数的一半，否则容易卡顿 |

**用法示例**

**场景一：单模块独立发版前的全域验证**

- **适用时机：** 当前迭代仅负责 `cad` 模块，需确保对 `cad` 模块内部头文件的清理，不会引发外部（如 `cam` 模块）的编译中断。
- **执行逻辑：** 仅提取大表中的 `cad` 模块数据，对其进行二分隔离测试。必须指定顶层 `zw3d.sln` 以验证跨模块影响。
- **命令配置：**

> ```
> python batch_rebuild.py --csv build_passed.csv -s "D:\ZW3D\out\build\zw3d.sln" --module "cad" --max-cpu-count 10 
> ```

- **预期行为：** 仅针对 `cad` 的更改引发多次顶层重构，最终产出带有 `Y/N` 标记的 `cad` 模块专属质检报告。

**场景二：多耦合模块联合二分排查**

- **适用时机：** 同时重构了底层基础库（如 `math_core`）与上层业务库（如 `cad`），需联合排查两者间的依赖断裂点。
- **执行逻辑：** 提取指定多个模块的数据置于同一个候选集合中，启动交叉二分搜索。
- **命令配置：**

> ```
> python batch_rebuild.py --csv build_passed.csv -s "D:\ZW3D\out\build\zw3d.sln" --module "math_core,cad" --max-cpu-count 10 
> ```

**场景三：自动化夜间流水线**

- **适用时机：** 服务器夜间挂机，或版本封板前的全量代码规范化阶段。对物理时间不敏感，追求绝对的排查覆盖率。
- **执行逻辑：** 无视模块边界，将数百个待测项纳入全局二分池。
- **命令配置：**

> ```
> python batch_rebuild.py --csv build_passed.csv -s "D:\ZW3D\out\build\zw3d.sln" --max-cpu-count 10 
> ```

- **预期行为：** 脚本可能耗费数小时至十数小时，在多次全量 Rebuild 与二分回滚中，最终在物理层和数据层同步锁定所有可安全移除的冗余代码。次日清晨可直接基于输出进行 Git Commit 提交。

##### 4. `compare.py` (报告差异比对工具)

- **定位：** 数据分析与回归测试辅助工具。

- **工作原理：** 用于比较两个 CSV 文件（例如不同分支的扫描结果，或两次不同扫描引擎的产出）中相同和不同的头文件记录。

- **用途：** 帮助研发人员追踪头文件依赖的恶化或优化趋势，或找出引发冲突的交集文件。

- **用法示例：**

  ```
  python compare.py <csv_file1> <csv_file2>
  ```

###  推荐工作流

根据时间预算和验证范围，强烈建议采用以下两种工作流之一：

#### 工作流 A：极速双漏斗流（日常开发/小范围重构 首选）

**耗时极短，兼顾精准与效率。**

1. **第一波过滤：** 运行 `fast_build.py` 接收原始扫描结果，利用增量编译极速剔除低级误报，生成 `build_passed.csv`。
2. **第二波验证：** 将 `build_passed.csv` 喂给 `batch_rebuild.py`，执行一次批量重构。
3. **人工收尾：** 若批量重构失败，脚本会自动复原代码。此时只需打开 Visual Studio，通过报错日志找出剩余那几个顽固的“隐式依赖”头文件，从 CSV 中剔除后再次运行步骤 2 即可。

#### 工作流 B：周末全量流

**耗时极长，但结果最权威，零人工干预。**

- 在周五下班前，直接将数千条记录的 CSV 喂给 `super_build.py`。
- 脚本会彻夜运行四象限交叉验证，周一早晨直接查收带有 `Y/N` 终极标识的 CSV 报告。

### 注意事项与环境配置

1. **多核并发配置：** 脚本内部已强制开启 `/m` 与 `/p:MultiProcessorCompilation=true` 以榨干 CPU。在执行测试期间，电脑可能出现轻微卡顿，属正常现象。
2. **MSBuild 路径：** 默认配置为 Visual Studio 2022 的标准安装路径。若您的环境为 VS2019 或装在非 C 盘，请通过 `--msbuild` 参数显式指定。
3. **文件锁问题：** 脚本内置了 `kill_vs_processes` 函数用于在每次编译前清理驻留的 `VBCSCompiler.exe` 和 `mspdbsrv.exe`，请勿手动删除该逻辑，否则会导致多核编译时 PDB 符号文件锁死。
